## Pre-Analysis & Key Decisions

### Detected Ambiguities (require caller confirmation)

1. **HKDF `info` vs `version` (brief §3.1 vs TODO signature).**
   - Brief literally: `info = "cobranza-encryption-v1:${keyName}"` (no version).
   - TODO signature: `deriveKey(masterKey, keyName, version?)`.
   - Brief §7 Key Rotation: "Library must decrypt any historical `version`" → `version` must produce
     distinct derived keys, otherwise rotating with the same `masterKey` is impossible.
   - **Recommended resolution**: include version in `info` when provided.
     - Default (no version): `info = "cobranza-encryption-v1:${keyName}"` (brief-literal, rotation v1).
     - With version N: `info = "cobranza-encryption-v1:${keyName}:vN"`.
   - This honors the brief default AND makes the TODO `version` param meaningful. The default case
     without version reproduces the brief exactly.

2. **max-arguments-per-method rule (≤2 params) vs TODO 3-param signature.**
   - `deriveKey(masterKey, keyName, version?)` has 3 params → violates `.kilo/rules/max-arguments-per-method.md`.
   - Rule: encapsulate >2 params in an object type defined in a new file.
   - `hkdf.ts` is an **internal** module (not public API — architecture.md: "Internal HKDF
     derivation"), so signature change is safe and does not alter the public API surface.
   - **Recommended resolution**: introduce `DeriveKeyParams` type in new file `src/hkdf.types.ts`
     and signature `deriveKey(params: DeriveKeyParams): Buffer`.
   - Alternative (TODO-literal 3-param) violates an enforced project rule. Not recommended.

### Technical / Architecture Decisions

- Node.js built-in `crypto.hkdfSync(digest, ikm, salt, info, keylen)` returns `ArrayBuffer`; convert to
  `Buffer` via `Buffer.from(arrayBuffer)`. Available since Node v15+; project targets Node 22.14.0.
- `digest` = `"sha256"` (HKDF-SHA256 per brief §3.1).
- `keylen` = 32 bytes (AES-256 key).
- `salt` = empty `Buffer` when not provided (brief: "optional fixed salt or empty").
- `masterKey` is base64-encoded; decode to raw bytes and assert length === 32 before derivation.
- Error strategy = "fail closed": throw clear, non-sensitive `Error` messages; never leak key bytes.
- No `process.env` reads (architecture.md Security Boundaries).
- Private members by default; `deriveKey` is module-scoped export for internal use.
- JSDoc on all exported symbols with `@example` blocks.

### Exact code

```ts
/**
 * Internal HKDF-SHA256 key derivation.
 *
 * Derives per-category 32-byte AES-256 keys from a single base64-encoded 32-byte master key
 * using Node.js `crypto.hkdfSync` (brief §3.1).
 *
 * HKDF parameters:
 * - `digest`: "sha256"
 * - `salt`: empty buffer (no fixed salt by default)
 * - `info`: "cobranza-encryption-v1:${keyName}" (+ ":vN" when `version` is provided)
 * - `keyLength`: 32 bytes
 */

import { hkdfSync } from 'node:crypto';

import type { DeriveKeyParams } from './hkdf.types.js';

const HKDF_DIGEST = 'sha256';
const HKDF_INFO_PREFIX = 'cobranza-encryption-v1';
const DERIVED_KEY_LENGTH_BYTES = 32;
const MASTER_KEY_LENGTH_BYTES = 32;
const EMPTY_SALT = Buffer.alloc(0);

/**
 * Build the HKDF `info` buffer for a given key name and optional version.
 */
function buildHkdfInfo(keyName: string, version?: number): Buffer {
  const versionSuffix = version === undefined ? '' : `:v${version}`;
  const info = `${HKDF_INFO_PREFIX}:${keyName}${versionSuffix}`;
  return Buffer.from(info, 'utf8');
}

/**
 * Decode a base64 master key and assert it is exactly 32 bytes.
 */
function decodeMasterKey(masterKeyBase64: string): Buffer {
  const masterKeyBuffer = Buffer.from(masterKeyBase64, 'base64');
  if (masterKeyBuffer.length !== MASTER_KEY_LENGTH_BYTES) {
    throw new Error(
      `Invalid masterKey: expected ${MASTER_KEY_LENGTH_BYTES} bytes after base64 decode, ` +
        `got ${masterKeyBuffer.length} bytes.`,
    );
  }
  return masterKeyBuffer;
}

/**
 * Derive a 32-byte per-category AES-256 key from the master key via HKDF-SHA256.
 *
 * @example
 * const key = deriveKey({
 *   masterKey: 'base64...',
 *   keyName: EncryptionKey.PII,
 *   version: 1,
 * });
 * // key.length === 32
 *
 * @throws {Error} when `masterKey` does not decode to exactly 32 bytes.
 * @throws {Error} when `keyName` is empty.
 */
export function deriveKey(params: DeriveKeyParams): Buffer {
  const { masterKey, keyName, version } = params;
  if (!keyName) {
    throw new Error('Invalid keyName: must be a non-empty string.');
  }
  const inputKeyMaterial = decodeMasterKey(masterKey);
  const info = buildHkdfInfo(keyName, version);
  const derived = hkdfSync(
    HKDF_DIGEST,
    inputKeyMaterial,
    EMPTY_SALT,
    info,
    DERIVED_KEY_LENGTH_BYTES,
  );
  return Buffer.from(derived);
}
```

### Verification criteria

- `deriveKey({ masterKey: <32-byte base64>, keyName: 'pii' })` returns a `Buffer` of length 32.
- Deterministic: same inputs → same output bytes (HKDF is deterministic).
- Passing the same `keyName` with different `version` produces different outputs.
- Passing a 16-byte (or any ≠32) base64 master key throws an `Error` mentioning expected 32 bytes.
- Passing `keyName: ''` throws an `Error`.
- No console logging; no `process.env` access.

### Notes on rule compliance

- `deriveKey(params: DeriveKeyParams)` — 1 parameter → complies with max-arguments-per-method.
- `hkdfSync(...)` receives 5 args (Node.js built-in API; the rule targets project code, not vendored API signatures — immutability of an external API is unavoidable).
- Max nesting depth in `deriveKey` ≤ 2 (the throw guards are 1 level; the call to `hkdfSync` is at the same level).

```ts
/**
 * Helper utilities for `@cobranza-apps/crypto`.
 *
 * Shared low-level helpers used across the library: base64 (de)serialization, random IV generation,
 * buffer concatenation, and constant-time comparison wrappers for hash verification.
 */

import { randomBytes, timingSafeEqual } from 'node:crypto';

const IV_LENGTH_BYTES = 12;
const EQUAL_LENGTH_MATCH = 0;

/**
 * Decode a base64 string into a `Buffer`.
 *
 * @throws {Error} when `value` is empty or not valid base64.
 */
export function base64ToBuffer(value: string): Buffer {
  if (!value) {
    throw new Error('Invalid base64 input: expected a non-empty string.');
  }
  return Buffer.from(value, 'base64');
}

/**
 * Encode a `Buffer` as a base64 string.
 */
export function bufferToBase64(buffer: Buffer): string {
  return buffer.toString('base64');
}

/**
 * Generate a cryptographically strong random IV (default 12 bytes for AES-256-GCM).
 */
export function generateIv(byteLength: number = IV_LENGTH_BYTES): Buffer {
  return randomBytes(byteLength);
}

/**
 * Concatenate multiple buffers into a single buffer (used for `IV + ciphertext + authTag`).
 */
export function concatBuffers(...buffers: Buffer[]): Buffer {
  return Buffer.concat(buffers);
}

/**
 * Constant-time comparison of two base64 strings.
 *
 * Returns `false` for differing lengths without invoking `crypto.timingSafeEqual` (which would
 * throw on length mismatch). Safe for hash verification (`verifyHash`) per brief §3.2.
 */
export function constantTimeCompare(a: string, b: string): boolean {
  if (a.length !== b.length) {
    return false;
  }
  const aBuffer = Buffer.from(a, 'utf8');
  const bBuffer = Buffer.from(b, 'utf8');
  return timingSafeEqual(aBuffer, bBuffer) === EQUAL_LENGTH_MATCH;
}
```

### Verification criteria

- `base64ToBuffer('SGVsbG8=')` decodes to bytes of ASCII "Hello".
- `bufferToBase64(Buffer.from('Hello'))` returns `'SGVsbG8='`.
- `generateIv().length === 12`; `generateIv(16).length === 16`; two calls produce distinct outputs.
- `concatBuffers(Buffer.from('a'), Buffer.from('b'))` equals `Buffer.from('ab')`.
- `constantTimeCompare('abc', 'abc') === true`; `constantTimeCompare('abc', 'abd') === false`;
  `constantTimeCompare('abc', 'abcd') === false` (no throw on length mismatch).
- No `process.env`, no logging.

### Notes on rule compliance

- All exported functions have ≤ 2 parameters (`generateIv(byteLength)` is 1; default provided).
- `concatBuffers` uses a rest parameter (counts as one parameter slot) — complies.
- Max nesting depth ≤ 1 in every function body.

## Risk / Edge Cases Considered

- `exactOptionalPropertyTypes: true` — `CryptoConfig.currentVersion?: number` means `undefined` only;
  the interface above uses `readonly currentVersion?: number;` which is correct under this flag.
- `noUncheckedIndexedAccess: true` — no indexed access is used in any proposed file.
- `Buffer.from(base64Str, 'base64')` does NOT throw on invalid base64; it silently drops invalid
  chars. `deriveKey` defends via the explicit 32-byte length assertion (invalid base64 → wrong byte
  length → assertion throw). `base64ToBuffer` does not assert length (callers like hashSalt may be
  >=32 bytes; length validation belongs to the consumer, e.g. `SecureCrypto` constructor in Task 3).
- `crypto.timingSafeEqual` throws `RangeError` when buffers differ in length — `constantTimeCompare`
  guards with an early length check and converts both strings to same-sized buffers only after.
- HKDF `salt` empty buffer is valid per RFC 5869 (defaults to a string of HashLen zeros).
