## Issues Found

### 1. JSDoc parameter names do not match actual parameter names (medium)

The stub method parameters were prefixed with `_` to satisfy `noUnusedParameters` / ESLint's `argsIgnorePattern: "^_"`, but the JSDoc `@param` tags still reference the original names.

Examples:

```ts
/**
 * @param plaintext - Plaintext to encrypt.
 * @param keyName - Logical key category ...
 */
encrypt(_plaintext: string, _keyName: EncryptionKey | string): EncryptedValue {
```

This produces stale hover/docs and violates the self-documenting-code rule.

**Fix:** Align JSDoc `@param` names with the actual parameter names in all five stub methods:
- `encrypt`: `_plaintext`, `_keyName`
- `decrypt`: `_encryptedValue`
- `hash`: `_plaintext`
- `verifyHash`: `_plaintext`, `_expectedHash`
- `encryptAndHash`: `_plaintext`, `_keyName`

Alternative (preferred for public API): remove the `_` prefixes and add a `void <param>;` statement at the top of each stub to mark the parameter as used, then keep the original JSDoc names. Either option is acceptable as long as names match exactly.

---

### 2. Constructor does not guard against `null`/`undefined` config (low/medium)

The JSDoc states:

```ts
@throws {Error} when `config` is null/undefined.
```

but the constructor immediately dereferences `config.masterKey`, which would throw a `TypeError`, not an `Error`.

**Fix:** Add an explicit guard at the start of `resolveConfig` (or the constructor):

```ts
if (!config) {
  throw new Error('Invalid config: expected a CryptoConfig object.');
}
```

This matches the documented behavior and avoids leaking a runtime `TypeError` to consumers.
