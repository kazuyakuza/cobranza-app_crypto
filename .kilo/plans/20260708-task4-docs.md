`encrypt`, `decrypt`, `hash`, `verifyHash`, `encryptAndHash`, `hasKey`,
`getAvailableKeys`, `destroy`.
- `encrypt(plaintext, keyName): EncryptedValue` → returns
  `{ encryptedData, keyName, algorithm: 'aes-256-gcm', version }`.

- `hash(plaintext): string` — base64 HMAC-SHA256 keyed by `hashSalt`.

- `encryptAndHash(plaintext, keyName): { encrypted: EncryptedValue; hash: string }`.
- `EncryptedValue` lives in `@cobranza-apps/entities`:
  `{ encryptedData, keyName, algorithm?, version? }`.
- Output payload layout: base64 of `IV(12) + ciphertext + authTag(16)`.

#### 3d — Add a new "Performance Notes" section
Insert a new `## Performance Notes` section immediately after the
`## Key Rotation Procedure` section (before `## Testing`). Content:
- Note: the library caches **derived** per-category keys in memory (HKDF output)
  keyed by `${keyName}:v${version}`, so repeated `encrypt`/`decrypt` calls do not
  re-derive; construct `SecureCrypto` once and reuse the instance.
- `decrypt` is cheap (AES-256-GCM) but for hot read paths, consumers MAY cache
  decrypted values in-memory with a **short TTL** — only when the cache is
  isolated per request/process and not shared across users/tenants.
- Do NOT cache plaintext in shared/observable caches; prefer a keyed in-memory
  LRU with size cap and clear on key rotation.
- Invalidate cached plaintext whenever the underlying `version` rotates or the
  secret is reloaded; `crypto.destroy()` clears the internal derivation cache
  when tearing down an instance.
- Hashing (`hash`/`verifyHash`) is deterministic and idempotent; safe to call
  repeatedly without caching.
- For bulk re-encryption during rotation, run in an external background job with
  batching/rate-limiting to avoid saturating event-loop latency (brief §7).
