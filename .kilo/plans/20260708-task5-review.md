### 2.3 Security concerns — ONE low-severity hardening opportunity

**Finding:** `SecureCrypto.destroy()` in `src/crypto.service.ts` zeros cached derived keys but does **not** zero the decoded `hashSaltBytes` buffer.

```ts
// src/crypto.service.ts
destroy(): void {
  for (const key of this.derivedKeysCache.values()) {
    key.fill(0);
  }
  this.derivedKeysCache.clear();
  // hashSaltBytes is not zeroed
}
```

Impact: low. The instance is documented as "must not be used after calling", so the salt buffer remains in memory until GC runs. For defense-in-depth, the salt buffer should also be overwritten.

The base64 master key inside `this.resolvedConfig` is an immutable JavaScript string and cannot be reliably zeroed, so no action is proposed there (consistent with prior plan decisions).

No other security issues were identified:
- AES-256-GCM uses random 12-byte IVs (`crypto.randomBytes`).
- HMAC-SHA256 verification uses `crypto.timingSafeEqual` (length mismatch returns early, but HMAC output length is fixed at 32 bytes / 44 base64 chars, so this is not exploitable here).
- No direct `process.env` reads inside runtime code.
- No `console.log`, `eval`, `Math.random`, or weak hash primitives.
